# /cm

**역할:** 커밋 메시지 규칙에 따라 적절한 메시지를 제안하고 사용자의 확인을 받습니다.

**사용 시점:** 코드 수정, 문서 업데이트, 리팩토링 등 작업 단위를 완료하고 커밋하기 직전

## 커밋 메시지 규칙

커밋 메시지는 다음 구조를 따릅니다:
`type: description`

### Types (유형)
- **feat**: 새로운 기능 요구 사항 구현 (예: 신규 페이지, API 연동 등)
- **fix**: 잘못된 코드 논리, 버그, 오타 등으로 발생하는 문제 해결
- **docs**: README.md, AGENTS.md, 코드 내 주석 등 문서 관련 변경
- **style**: 코드 의미에 영향을 주지 않는 수정 (들여쓰기, 세미콜론, 개행 등)
- **refactor**: 버그 수정이나 기능 추가 없이 성능 개선 또는 구조 변경
- **test**: 테스트 코드 작성, 리팩토링 및 누락된 테스트 추가
- **chore**: 의존성 패키지 관리, 환경 설정, 빌드 스크립트 수정 등 프로젝트 관리성 작업

### 가이드라인
1. **언어**: 커밋 메시지의 `description`은 **한국어**로 작성합니다.
2. **명확성**: 변경 사항을 한눈에 알 수 있도록 간결하고 명확하게 작성합니다.
3. **현재 시형**: '추가함', '수정함'과 같은 완료형보다는 '추가', '수정'과 같은 명사형 또는 현재형을 권장합니다.

## 워크플로우 단계

1. **변경 사항 분석**: 작업이 완료된 후 변경된 파일과 내용을 검토합니다.
2. **메시지 생성**: 위의 규칙에 따라 적절한 `type`과 `description`을 조합하여 메시지를 생성합니다.
3. **사용자 확인**: 생성된 메시지를 사용자에게 제안합니다.
4. **실행**: 사용자가 승인하면 커밋 명령을 준비하거나 직접 실행합니다.

---
**예시:**
- `feat: 에이전트 워크플로우 설정 추가`
- `docs: AGENTS.md 지침 업데이트`
- `fix: 오타 수정 및 경로 오류 해결`
